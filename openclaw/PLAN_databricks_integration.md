## Technical Requirements

### Delta Lake Integration
- Use `databricks-sql-python` to connect to Databricks SQL Warehouse
- Log research summaries directly into a **Delta Table (Gold Layer)**
- Table location: Unity Catalog (fully qualified: `catalog.schema.table`)

### Cryptographic Integrity
- Before every insert, generate a **SHA-256 hash** of the research entry content
- Store the hash as a column in the Delta Table — creates an immutable audit trail of the agent's findings

### Unity Catalog Governance Query
- Agent must be able to query **Unity Catalog Information Schema** (`information_schema.tables`, `information_schema.column_tags`, `information_schema.table_privileges`)
- Use these queries to identify and summarize: schema changes, permission changes on tables

### Human-in-the-Loop Hook
- All new entries written by the agent must have **`status = 'DRAFT'`**
- The human (David) manually reviews and runs a promotion query to set `status = 'PUBLISHED'` for entries to enter the final archive
- The Delta Table schema must include: `status VARCHAR` with allowed values `DRAFT` / `PUBLISHED`

## Architecture Constraints

### Zero-Trust Authentication
- **Service Principal only** — no standard user login (this is a headless agent)
- All credentials loaded from environment variables, never hardcoded:
  - `DATABRICKS_HOST` — workspace URL (e.g. `https://adb-xxxx.azuredatabricks.net`)
  - `DATABRICKS_TOKEN` — Service Principal OAuth token or PAT
  - `DATABRICKS_HTTP_PATH` — SQL Warehouse HTTP path
  - `DATABRICKS_CATALOG` — target Unity Catalog name
  - `DATABRICKS_SCHEMA` — target schema name
  - `DATABRICKS_TABLE` — target Delta Table name
- These must be added to the `.env` template in `init_agency.sh`

### Isolation & Volume Mount
- Code runs inside the existing `openclaw-hardened` container — no new container
- Local write buffer for offline/failed writes must go to:
  **`/home/clawuser/workspace/research_logs/`**
  (which maps to the host's `/mnt/disks/research/workspace/research_logs/` on the 50GB persistent disk)
- If the Databricks connection fails, the worker must write to this local buffer and retry on next run — no data loss on connection blips

### Code Structure
- **Modular, object-oriented Python**
- At minimum, separate classes for:
  - `DatabricksConnector` — handles connection, retries, SQL execution
  - `ResearchEntry` — data model (content, hash, status, timestamps, metadata)
  - `AuditHasher` — SHA-256 generation and verification
  - `CatalogGovernanceChecker` — Unity Catalog metadata queries
  - `DatabricksWorker` — orchestrates the above, called by the agent

## Delta Table Schema

The Gold Layer table should have at minimum:

| Column | Type | Notes |
|---|---|---|
| `entry_id` | STRING | UUID, generated by agent |
| `created_at` | TIMESTAMP | UTC insert time |
| `topic` | STRING | Research topic label |
| `summary` | STRING | Full research summary text |
| `content_hash` | STRING | SHA-256 of `summary` — immutable audit trail |
| `status` | STRING | `DRAFT` (agent-written) or `PUBLISHED` (human-approved) |
| `source_agent` | STRING | Agent identifier (e.g. `openclaw-v4`) |
| `metadata` | MAP<STRING,STRING> | Flexible key-value for extra context |
